Overview

In this homework, I implemented a full path tracer that simulates global illumination using Monte Carlo integration. Starting from basic ray generation, I built a rendering pipeline that traces rays from the camera into the scene, computes intersections with geometry, and estimates radiance using both direct and indirect lighting. I implemented diffuse BSDF sampling, recursive multi-bounce light transport, and Russian roulette for unbiased termination of light paths. Additionally, I incorporated adaptive sampling to dynamically allocate more samples to noisy pixels while stopping early for pixels that have converged. One interesting learning was the process of light transport. For example, surfaces that receive no direct light, such as the ceiling, can still become brightly illuminated due to indirect light bouncing from other surfaces. This assignment helps me build a solid understanding of ray-tracing and illumination.

Part 1: Ray Generation and Scene Intersection

Implementation of the ray generation pipeline: To generate camera rays from image coordinates, I did the following three step in camera/generate_ray() function:

  1. Calculate the coordinates in the virtual camera sensor using linear interpolatoin -- \( x\_camera = - tan(hFov\_radian / 2) + x * (2 * tan(hFov\_radian / 2)) \), similar for y_camera.
  2. After I got the camera coordinates, I obtained the direction of the ray in camera space from the origin simply as (x_camera, y_camera, z_camera), where z_camera is -1 in this case.
  3. Based on the direction of ray in camera space, I calculated the direction of the ray in world space by transforming the direction vector by the c2w matrix. The ray origin in the world space is the camera's position
Then, in the pathtracer/raytrace_pixel() function, I used Monte Carlo Algorithm to get the radiance value for each pixel by taking ns_aa samples inside the pixel. I used gridSampler->get_sample() helper function to generate a uniformly random Vector2D sample with values from 0 to 1. Then, I added this generated sample to the original (x, y) position to get the position of the sample that's inside the pixel. I found the ray at the coordinates using the generate_ray() function that I implemented before, and then calculated the radiance in est_radiance_global_illumination(). Finally, I took the average radiance across all samples to get the return radiance value of the given pixel.

Implementation of ray - primitive intersection: Generally speaking, for the intersection with triangle and sphere, I used the formula shared during the lecture on how to calculate the intersection time. For ray-triangle intersection, the function is \[ dot(p^1 - ray.o, plane\_normal)\ /\ dot (ray.d, plane\_normal) \], and for ray-sphere intersection, the function is \[ (-b +- sqrt(b^2 - 4ac)) / 2a \] where a = \( dot(r.d, r.d) \), b = \( 2.0 * dot(r.o - o, r.d) \), and c = \(dot(r.o-o, r.o-o) - r2 \). After I found the time, I checked whether it's valid or not. If it is valid, I concluded that there was an intersection and set up the intersection structure.

Triangle intersection algorithm: I implemented ray triangle intersection in two steps:

  1. I computed two edge vectors from the triangle vertices and used their cross product to get the plane normal. Then I solved for the parameter t of the ray equation to find where the ray hits the plane using the formula from lecture \( dot(p^1 - ray.o, plane\_normal)\ /\ dot (ray.d, plane\_normal) \). If t is valid (>= 0 and < infinity), the ray intersects with the plane. Thus, I computed the hit point by plugging t back into the ray equation.
  2. To determine whether this point lies inside the triangle, I used edge tests. For each edge, I computed the cross product between the edge vector and the vector from the edge start to the hit point. If all these cross products have the same orientation relative to the triangle normal, then the intersection point is inside the triangle. In that case I updated the ray.max_t to t so that farther intersections can be ignored and returned true. Otherwise I returned false.
I did similar things for the intersect() function with some additional steps that sets the intersection structure as instructed.

Part 2: Bounding Volume Hierarchy

BVH construction algorithm: I constructed the BVH recursively. For each call to construct bvh, I first looped through all primitives in the current range and expanded a bounding box so that it enclosed every primitive in that range. I then created a BVH node with this bounding box. If the number of primitives in the range was less than or equal to max_leaf_size, I made the node a leaf by storing the start and end iterators and returned it directly. Otherwise, I split the primitives into two smaller groups and recursively built the left and right child nodes. To decide how to split, I computed a second bounding box over the centroids of the primitive bounding boxes rather than over the full primitive boxes. I used this centroid bounding box to measure how spread out the primitives were along each coordinate axis. I then chose the axis with the largest extent to be splitted in the mid point.

The heuristic behind my splitting algorithm is that a greater axis spread indicates that the bounding boxes are more likely to be separated, which would make BVH more efficient by reducing overlap of child nodes. Also, picking the mid value will make the primitives more equally spreaded. The figure below demonstrates how BVH splits the primitives.

Comparison of rendering time: BVH acceleration significantly reduces the rendering time. For example, in one trial, the average number of rays per second for the "maxplank" without BVH was 0.0069 million, while that with BVH acceleration had an average of 3.1766 million rays per second. Moreover, this performance gain becomes more pronounced as scene complexity grows.

Part 3: Direct Illumination

Implementation of direct lighting functions

  1. Direct lighting with uniform hemisphere sampling: This function samples light uniformly in all directions in the hemisphere. I estimated how much light arrived at the intersection point (hit_p) by integrating over all light arriving in a hemisphere around that point. I approximated the integral using the Monte Carlo Estimator with the equation provided in lecture \[ \frac{1}{N}\sum_{j=1}^{N}\frac{f_r\!\left(p,\omega_j \to \omega_r\right)\,L_i\!\left(p,\omega_j\right)\cos\theta_j}{p(\omega_j)} \] To achieve this, I used a for loop with num_samples iterations. In each loop, I randomly sampled a w_in (direction of incoming light in local space), and converted it to the w_in_world (direction of incoming light in world space). With a hit_p and w_in_world, I created the ray to test whether it would hit a lighting source. Due to floating point precision, I set ray.min_t to EPS_F to prevent the ray from hitting the same triangle that it came from. Then, I traced the ray from the hit point in the sampled direction and found out its first intersection by calling bvh -> intersect (ray, &light_isect) so that the information of the first intersection is stored in light_isect structure. I checked the emission of the first intersecting object with L_in = light_isect.bdsf->get_emission(). If the value of L_in is greater than 0, it indicates that the object is a light source -- I would add its contribution to the total L_out (outgoing radiance) using the Monte Carlo Approximation mentioned before.

  2. Direct lighting with importance sampling lights: Instead of randomly sampling in the whole hemisphere, I wanted to account for point light and reduce noise level by only focusing on the light sources in the scene. To achieve this, I used a for loop to iterate over all light sources in the scene and determined the number of samples needed based on whether the light source is a point light or not. If it is a point light, only one sample is needed, else, I used ns_area_light samples. For each light source, I initialized a light_sum to store the total contribution of this light source across all samples. For each sample, I generated one incoming radiance from the light by calling light->sample(hit_p, &w_in_world, &dist_to_light, &pdf). Then, I converted the incoming direction (w_in_world) to w_in in local frame, and checked whether cos_theta(w_in) <= 0. If cos_theta(w_in) <= 0, it means the light is behind the surface and I would no longer consider it when calculating outgoing radiance. Then, I created the ray to test whether there's anything blocking between the hit point and the light source. To do so, I set ray.max_t to dist_to_light - EPS_F. Finally, I checked whether there's intersection between the hit point along the light direction that's in front of the light source, if there isn't, I would add the contribution to the total L_out using the same Monte Carlo Approximation algorithm.


Comparison of uniform hemisphere sampling and lighting sampling: The figures in the first row below are images rendered with uniform sampling, and the figures in the second row are images rendered with light sampling. The captions of the figures document -t: number of threads, -s: number of camera rays per pixel, -l: number of samples per area light, -m: maximum ray depth, -H: whether it's uniform hemisphere sampling. Comparing the rendered images, uniform hemisphere sampling produces significantly noisier results compared to lighting (importance) sampling, especially in regions with soft shadows. This is because hemisphere sampling distributes rays uniformly over all directions, causing most samples to contribute little or no light since they do not intersect light sources, leading to high variance and slow convergence. In contrast, lighting sampling directly targets the light sources, ensuring that most samples contribute meaningful illumination. As a result, lighting sampling generates much smoother shadows and cleaner images with far fewer samples, and it is also capable of correctly handling point lights, which hemisphere sampling almost never captures due to their infinitesimal size.


Noise levels in soft shadows when rendering with different number of light rays: Below are figures rendered with different number of light rays (-l) and with 1 sample per pixel (-s 1) under light sampling. A soft shadow is a shadow with gradual and blurry edges. In Monte Carlo integration, each light ray is a sample (ns_area_light). Thus, if we have fewer light rays / samples, each random sample ray has a larger influence on the final average, so the estimate fluctuates more, producing visible noise -- For example, a pixel may have all of its sampled rays blocked even though the true lighting at that point is only partially occluded, leading to an underestimate of the incoming light. As the number of light rays increases, the average over these samples becomes more stable and better approximate the true lighting. However, with larger number of light rays, the rendering time also increases significantly.

Part 4: Global Illumination

Implementation of indirect lighting: For indirect lighting, I implemented at_least_one_bounce_radiance() as a recursive function that estimates radiance arriving at a surface from one or more additional bounces. I initialized the ray depth to max_ray_depth.

  1. Base case: if ray.depth == 0, I return L_out, which is initialized to (0, 0, 0), immediately because no reflected light is added and only the zero_bounce emission is visible.
  2. Recursive step: if isAccumBounces == true, I add one_bounce_radiance(r, isect) at every recursion so that the result contains all lighting contributions from bounce 1 up to the maximum depth. Else if isAccumBounces == false, I only add one_bounce_radiance(r, isect) when r.depth == 1, which corresponds to the final allowed bounce and makes the renderer isolate only the light contribution from the selected bounce depth instead of summing all previous bounces.
To estimate higher-order indirect lighting, I sample a new incoming direction from the surface BSDF by calling the sample_f (w_out, &w_in, &pdf) function, which writes to w_in and pdf. I then detect invalid samples by checking whether pdf <= 0 or cos_theta(w_in) <= 0 (behind the surface), if these are the cases, I disregard the sample. Because the sampled direction from BSDF is in local space, I convert it back to world space and then spawn a new ray from the hit point in that direction called next_ray with depth decremented by 1 from the current ray. If the next_ray intersects another object, I recursively call at_least_one_bounce_radiance(next_ray, next_isect) to estimate the radiance arriving at the next surface from deeper bounces. I store this result in L_out_next. Finally, I combine this recursive estimate differently depending on whether we want to accumulate bounces. In the accumulated mode, I add the L_out_next to L_out with the Monte Carlo Approximation in each recursive call (L_out += \((f * L\_out\_next * cos\_theta(w\_in))\ /\ pdf \)) so tha final result contains all bounce contributions. In the non-accumulated mode, I use L_out = \((f * L\_out\_next * cos\_theta(w\_in))\ /\ pdf \) so that earlier-bounce contributions are not preserved, allowing the renderer to visualize only the max_depth bounce of light.

Finally, in est_radiance_global_illumination(const Ray &r), When isAccumBounces is true, the renderer sums zero_bounce_radiance and at_least_one_bounce_radiance. When isAccumBounces is false, I check the max_ray_depth, if max_ray_depth is 0, I only return zero_bounce_radiance, else, I only return at_least_one_bounce_radiance.

Part 5: Adaptive Sampling

Implementation: For each pixel, I trace multiple camera rays by sampling random subpixel locations and accumulating their radiance contributions. In addition to summing radiance, I maintain two running statistics:

  1. s1: sum of sample illuminance
  2. s2: sum of squared sample illuminance
These two variables are used in the calculation of mean and variance. Every SamplesPerBatch sample (size = i), I estimate the pixel's convergence by computing the mean and standard deviation and then forming a 95% confidence interval I = 1.96 * mean / varaince, where mean = \( s1\ /\ i\) and variance = \((s2 - (s1 * s1)\ /\ i)\ /\ (i - 1)\). If I <= MaxTolerance(0.05) * mean, I assume that the pixel has converged and stop tracing more rays for this pixel. If not, I would continue the tracing-and-detecting loop until reaching the maximum sample size.
